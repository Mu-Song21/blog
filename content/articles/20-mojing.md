## 一开始想清楚的问题

线下书画展的局限很明确：观众难以进入作品空间，也缺少创作与学习的延伸。竞赛与答辩场景又要求系统必须**现场可运行**，不能只停在概念演示。

因此「墨境」（产品全称墨境流芳）的目标不是堆砌页面，而是一条完整链路：

**观展 → 创作 → 学习 → 归档 → 运营**

作品集短名取「墨境」，与仓库名 `mojing-liufang` 对应。

## 系统架构

```
Vue 3 + TypeScript
  ├─ scene.ts（Three.js：山水 / 书法主馆）
  ├─ three-jl（⑤灵兽区独立 WebGL 上下文）
  ├─ 水墨工坊 / 国风课堂 / 我的空间 / 运营后台
  └─ WebSocket：工坊任务进度
        ↕
Spring Boot 3.2 + MyBatis-Plus + MySQL（开发环境可用 H2）
  ├─ 展品、展区、用户、收藏、访问统计
  └─ 异步生图任务（云端兼容 API 可选；本地水墨算法兜底）
```

Docker Compose 可编排 MySQL、Redis、EMQX 与前后端。本地默认关闭 Redis 与 MQTT，避免演示依赖额外中间件。

## 关键实现

### 三展区 Web3D

当前可演示展区为 **山水、书法、灵兽**；非遗与器物展区已下线，不应写成「六区完整交付」。主馆由较长的场景脚本负责行走、沉浸浏览与后处理；灵兽区使用独立 `three-jl` 粒子叙事，与主馆 WebGL 上下文隔离，降低资源冲突与故障扩散。

### 水墨工坊

流程为：上传 → Canvas 本地预览 → 后端异步生成 → WebSocket 回报进度 → 卷轴三维预览。配置了兼容生图 Key 时可走云端；未配置时由 Java 本地水墨处理出图，保证演示不中断。另支持程序化导出 GLB / STL（简化卷轴几何，非文物扫描建模）。

### 课堂与运营后台

课堂含四门课程与分步课纲，进度可保存在本地；运营后台支持展品维护、资源上传，以及基于真实 `visit_log` 的 ECharts 看板。这两块常被忽略，却最能体现全栈闭环是否成立。

## 能力边界（不宜夸大）

| 不宜声称 | 实际情况 |
|----------|----------|
| 讯飞星火已驱动智能赏析 | 演示主路径为预写策展文案 + 打字动效；相关接口未接入主流程 |
| 六展区全部上线 | 可演示为三区 |
| AR、小程序、文创商城与支付 | 本期未做 |
| Redis / MQTT 为生产刚需 | 可选能力，默认可关闭 |
| 完整 Spring Security | 轻量 Token 会话，适合演示环境 |

## 收获

Web3D 项目的难点往往不在「能否调通 Three.js」，而在资源隔离、性能分级，以及**无外网、无 API Key 时仍能完成演示**。这一点与物联网项目中的 mock 上报属于同一类工程取舍。
